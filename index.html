What is git?



Git is an open source version control system designed track changes files & meta information with speed & efficiency.

Also referred to as a Source Code Management, or SCM tool.


Does this sound familiar? Most likely you've heard of other source control management systems:

  • Subversion
  • Microsoft Team Foundation Server
  • Mercurial

Not all VCS's work the same

Git is a distributed version control system. This means full copies of the repository are on every contributing machine.

git | Individuals

Not just for teams!

Even if you are independently editing files by yourself & not working on a team, you can still take advantage of all the benefits of a version control system


Individual Benefits

  • Ability to undo changes & revert back to a previous snapshot
  • Confidence to make changes
  • Ability to create isolated environments for experimenting with new features
  • Write messages to your future self
  • Maintain your own preferred frequency of commits & work flow
  • If you make a mistake, you can always revert back to a stable point in the git history
  • If you decide to add collaborators to your project, catching them up will be an easy transition
  • A complete history of all the changes made
  • Complete documentation history of why changes were made
  • Ability to checkout a snapshot of a previous point in time (in an isolated environment) & inspect the code
  • Multiple History Streams

git | Teams

Team Benefits


Ability to resolve conflicts

With Git, multiple people can work on the same file at the same time. Usually git will be able to merge the changes automatically. If it can't, it'll show you what the conflicts are and will make it easy for you to resolve them.


Independent History Streams

Different people on the project can work on different branches, allowing you to work on separate features independently and then merge the features when they're done.

GitHub

Simply stated, GitHub is a web-based git repository hosting service.

GitHub combines revision control, source code management (SCM), & additional features all into one platform.

GitHub extends beyond the git command-line with a fully functioning graphical interface, both for desktop & mobile platforms.

GitHub was originally created as a way to make it easier for developers to contribute to open source projects.

More than open-source
In addition to open-source projects, GitHub is quickly becoming the default platform for software development

Source Code Management + Features
More than just a source code management tool, GitHub provides many additional tools for specifying, discussing, & reviewing software.

Installing git



If you don't have git installed yet, you can download the latest version git's website

Install git


Post-install

After you have downloaded & installed git on your system, open up the terminal & type in the following command:

$ git --version

It should return something like this:

$ git version 2.2.1

You can also see the path of your git installation by typing:

$ which git

It should return something like the example below

/usr/local/bin/git

Setup git (mandatory)

First-time git config setup

Now that you have git installed, we will need to configure your git user settings. When you initialize git to start tracking file changes, a hidden subdirectory named .git will be created at the root of your project. This is where git stores all the version history, user configuration settings, etc.

Luckily, git provides an amazing command-line tool that allows us to edit any configuration settings right from the command-line. This saves us from having to dig around hidden .git directories trying to manually make changes to our configuration settings.


Now that we have git properly installed, we can configure our user settings



git config (mandatory)

To check if you have an existing user:

$ git config --global user.name

If you don't have an existing user, create one now by running the same command along with the name of the user you want to create

$ git config --global user.name "John Ryan"

Next, let's check if we have an email associated with this user:

$ git config --global user.email

If there is no email address returned, go assign one now. 

$ git config --global user.email "your.name@domain.com"

Setup git (optional)

Optional git config setttings


To see all of your current git config settings for the logged in user:

$ git config --global --list

To see your system wide git config settings:

$ git config --list


Checking git config

$ git config --settings
$ git config --global
$ git config --user
$ git config --global --list
$ git config --list
$ git config --global color.ui
$ git config --global color.branch
$ git config --global color.diff
$ git config --global color.interactive
$ git config --global color.status
$ git config --global core.autocrlf


To change your git config settings you will need to pass in variable at the end of each command

Changing your config settings

$ git config --global color.ui true
$ git config --global core.editor emacs
$ git config --global core.editor vim
$ git config --global color.branch "red"
$ git config --global color.diff "blue black bold"
$ git config --global color.interactive
$ git config --global color.status
$ git config --global core.autocrlf

Remote Endpoints

When cloning an existing repo from github, a remote with an alias of origin that points to the url of the remote repository that was cloned.

In this case the remote will point to:

git@target.com:chef-training/git-clone-existing.git

which will be set automatically

Check for Existing Remotes

To see existing remotes set for any git repo type the following command:

Removing a remote

To remove a remote you can type git remote remove REMOTE-ALIAS, where REMOTE-ALIAS is the name of the remote you want to remove. The default remote from a cloned repo will have a name of origin.

Adding a remote

To add a remote you can run the command:

                        git remote add <REMOTE-ALIAS> <REMOTE-URL>

where REMOTE-NAME is the name of the remote you would like to add & REMOTE-URL is the endpoint that the remote will point to on GitHub. This could be origin, github, heroku, aws, docker, openstack, etc.

Changing a remote

To change the URL of an existing remote, run the command:

                        git remote set-url <REMOTE-ALIAS> <NEW-REMOTE-URL>

REMOTE-ALIAS - The name of the remote, whose endpoint you want to change
NEW-REMOTE-URL - The new URL you want to assign to this remote


In this example, origin is the alias of our remote & git@target.com:chef-training/git-clone-existing.git
is the remote url.

Example 1 - Staging files & typical workflow

Typical Workflow


Step 1 git add


git add is the command to move changes in your current working directory to the staging area.

Example:

Let's say that we have just created new index.html page & README.md & want to add them to the staging area for our next commit.

We can use the git add command along with the name of the files we want to add. In this case we would run the command:

$ git add index.html
$ git add README.md

Or to add all the files in your current directory (index.html + README.md), we can just run

$ git add *


Step 2 git status

Now that we have added files to the staging area, we can see what has been modified and staged since our last commit was made by running the git status command

$ git status A index.html A README.md

Step 3 git commit

Next we are going to the use the git commit command to record a snapshot of the staging area. Git records our name & email address from our git config --global user.name & git config --global user.email settings from earlier.

Along with our username & email address associated with these changes, you will want to also include a message that will also be included in this snapshot. To record your snapshot along with a commit message run the following command & provide a message inside quotations.

$ git commit -m "add index.html & README.md file"

If you forget to add the -m along with a message for the commit, git will automatically open the vim editor for you to add a commit message & save before continuing.



Step 4 git push

Now that we have created a commit, we can use the git push command to push (send) our changes to a remote repository.


To push our changes to a remote repo, we need to also include:

alias - name of the remote repository you want to send changes to branch
remote - name of the branch that you want to create (if not already created) on the remote repository 
$ git push [alias] [branch]

Common defaults

alias - origin
branch - master

If you are still working on the master branch & haven't changed any of the default settings after cloning, you would run the command:

$ git push origin master

About git init

The git init command adds a hidden .git subdirectory that stores all the meta information, file tracking, etc. You will not need to make any changes directly to this folder.

Instead, git provides you access to different configurations through their command-line tools.



git command-line cheat sheet

Although we don't need to go into the .git folder, we should check to make sure that its there & that git has been properly initialized. To check that you have properly initialized git for this directory, type in the command ls -la
